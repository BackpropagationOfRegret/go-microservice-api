Проект: Платформа заказа еды и доставки (а-ля Delivery Hero / Uber Eats)
Почему именно этот домен?

    Разные бизнес-процессы с разными требованиями к нагрузке и надёжности.

    Естественные границы сервисов (пользователи, рестораны, заказы, курьеры, платежи, уведомления).

    Много событий, которые нужно обрабатывать асинхронно.

    Возможность масштабировать отдельные сервисы — например, под Black Friday нагрузка на заказы растёт, а рестораны остаются прежними.

    Реалистичный размер для микросервисов: 5–7 сервисов, каждый может вести отдельная команда.

1. Доменная модель (ключевые сущности)

    Пользователь (клиент) — регистрация, профиль, история заказов.

    Ресторан — меню, статус (открыт/закрыт), адрес.

    Заказ — связь пользователя и ресторана, позиции, статус (создан, оплачен, готовится, передан курьеру, доставлен).

    Курьер — местоположение, статус, назначенные заказы.

    Платеж — сумма, метод, статус транзакции.

    Уведомление — SMS, push, email о статусе заказа.

2. Сервисы и их ответственность
Сервис	Обязанности	Основное взаимодействие
user-svc	CRUD пользователей, аутентификация (JWT), адреса доставки	Синхронно: gRPC/HTTP (проверка токена, получение профиля)
restaurant-svc	Управление ресторанами, меню, часами работы	Синхронно (для UI), асинхронно — события изменения меню (Kafka)
order-svc	Создание заказа, управление жизненным циклом, координация	Синхронно: API для клиентов. Асинхронно: публикует события OrderCreated, OrderStatusChanged
payment-svc	Обработка платежей (карты, кошелек), возвраты, чарджбеки	Асинхронно: получает событие OrderCreated, после успеха публикует PaymentProcessed
delivery-svc	Назначение курьеров, трекинг, статус доставки	Асинхронно: слушает OrderReady, назначает курьера, публикует CourierAssigned, OrderDelivered
notification-svc	Отправка уведомлений через каналы (email, SMS, push)	Асинхронно: подписывается на все ключевые события (заказ оплачен, курьер назначен, доставлен)
api-gateway (Traefik + агрегатор)	Единая точка входа для клиентов (мобилка/веб), агрегация данных из нескольких сервисов	Синхронно: HTTP/REST или GraphQL (BFF)

Дополнительно: event-store (опционально) или просто Kafka с долгим хранением.
3. Поток событий через Kafka (асинхронная магия)
Пример сценария: пользователь делает заказ

    Клиент → api-gateway → order-svc (синхронный POST /orders).

        order-svc проверяет через user-svc (синхронно gRPC) валидность пользователя.

        Запрашивает через restaurant-svc текущее меню и цены.

        Создаёт заказ в своей БД со статусом PENDING.

    order-svc публикует событие в Kafka:
    OrderCreated (order_id, user_id, restaurant_id, total_amount, items)

    payment-svc получает OrderCreated → пытается списать деньги (через платёжный шлюз).

        Если успех → публикует PaymentProcessed (order_id, transaction_id)

        Если ошибка → публикует PaymentFailed → order-svc отменит заказ

    order-svc получает PaymentProcessed → меняет статус на PAID → публикует OrderPaid.

    restaurant-svc подписан на OrderPaid → переводит заказ в свою "кухонную" систему (показывает повару). Когда заказ готов → публикует OrderReady.

    delivery-svc слушает OrderReady → находит свободного курьера (гео-поиск) → назначает → публикует CourierAssigned.

    notification-svc слушает ВСЕ события и отправляет пользователю push/SMS/email:
    OrderCreated → "Заказ принят"
    PaymentProcessed → "Оплата прошла"
    OrderReady → "Заказ готовится"
    CourierAssigned → "Курьер в пути"
    OrderDelivered → "Приятного аппетита!"

    delivery-svc после доставки публикует OrderDelivered, order-svc обновляет статус и завершает сагу.
